# Quizter Hosted Transformation Plan

## Purpose

This document captures the agreed plan for transforming Quizter from a
single-room LAN-hosted app into a simple public hosted service that can be run
from a managed platform.

The goal is to preserve the current game experience while removing the need for
the host to own a laptop. The first hosted version is intentionally simple and
avoids accounts, databases, public question authoring, and horizontal scaling.

## Product Direction

Quizter will move from:

- one local server process
- one global room
- one admin login
- LAN-based join flow

to:

- one public hosted site
- multiple private rooms inside one server process
- owner-token room control instead of admin login
- room-code and QR-based player join
- expiring rooms with lightweight resume

The first hosted version is optimized for the fastest credible prototype, with
a clean enough structure to support future persistence later if needed.

## Locked Product Decisions

### Public Site Model

- There will be one public homepage.
- Main homepage actions:
  - `Create Room`
  - `Resume Room`
  - `Join Room`
- Any visitor can create a room.
- Rooms are private unless the user knows the room code or player join link.
- Players join by room code and/or QR code.
- QR and share link are generated only after room setup is completed and the room
  is launched.
- Players cannot join until launch is complete.

### Room Identity And Ownership

- Room codes are 4-character alphanumeric codes.
- Codes are auto-generated by the server.
- If the server cannot generate a unique active code, room creation fails.
- Expired and closed room codes may be reused.
- Each room has exactly one owner.
- The owner is authenticated by a hidden room owner token.
- There are no admin passcodes.
- There are no user accounts in hosted v1.
- The owner token is tied only to the room, not to browser/device traits.
- The owner token remains hidden from the UI.
- Player join links must never contain the owner token.
- One browser may own only one active room at a time.
- If the browser already owns an active room, creating a new room should be
  blocked with a prompt to close the existing room.

### Resume Behavior

- A returning owner on the same browser should be able to resume the room if the
  room is still active and the owner token remains valid.
- The homepage should show `Resume Room` only when the browser has a valid owner
  token for an active room.
- Resume is explicit from the landing page, not an automatic redirect.

### Room Setup Wizard

- Room creation uses a guided wizard.
- Planned wizard flow:
  1. `Create Room`
  2. `Choose Question Packs`
  3. `Choose Game Settings`
  4. `Review / Launch Room`
- Room title is required before launch.
- Room title is for display only and is not used for joining.
- At least one question pack must be selected before launch.
- Rounds must be at least 1 and must not exceed the number of available
  questions from the selected packs.

### Question Bank Model

- Hosted v1 uses only server-side curated question packs.
- No public question creation.
- No public JSON upload/import.
- No per-user saved question libraries.
- The server should scan the question-pack directory when a new room is created.
- Each room receives a snapshot of the available pack library at creation time.
- Later filesystem changes should not affect existing rooms.
- Question pack selection is stored per room.
- Hosts may change selected packs dynamically while a game is in progress.
- Pack changes apply next round only.

### Room Lifecycle

- Rooms can exist through multiple games.
- Rooms expire after 30 minutes of no owner or player activity.
- `Close Room` immediately closes the room, removes players, invalidates the room
  code, and invalidates the owner token.
- `End Game` ends the current game but keeps the room open.
- After `End Game`, the host may adjust packs/settings and start a new game in
  the same room.
- If a game ends before any scoring has happened, players should see a simple
  host-ended message instead of a final leaderboard.

### Gameplay Behavior

- Hosts may change settings during a game.
- All dynamic changes should apply next round only where relevant.
- The host should be able to end the current game during an active round.
- Late joins are allowed after a game starts.
- Late joiners start with score 0.
- Late joiners must wait until the next question before answering.
- If the host disconnects briefly, the room and game continue.
- If the host returns with a valid owner token, control resumes seamlessly.
- Between games, players should see a waiting screen.
- Before a game starts, players should see a lobby showing room code and waiting
  status.
- Between games, players should not see a persistent leaderboard.

### Player Identity

- Player identity remains room code plus display name.
- Rejoining with the same display name in the same room should restore the same
  player identity and score when appropriate.
- Player names only need to be unique within a room.
- The host is not a player and does not have a player identity.

### Moderation

- The host can kick players.
- Kicked names are blocked from rejoining.
- Default behavior should be effectively permanent room-level blocking until
  manually cleared.
- There should be a configuration option for clearing kicked-name blocks on new
  game start.
- Default for that option should be off.
- The host can manually unban a blocked name while the room remains open.

### Deployment Direction

- First hosted deployment target is a managed platform.
- Current preference is DigitalOcean App Platform using repo/source deploy.
- Docker is not required for the first deployment.
- The app should still remain compatible with local binary execution.

## Non-Goals For Hosted v1

The first hosted version explicitly does not include:

- full user accounts
- password login
- email or magic-link auth
- database-backed persistence
- room recovery after server restart
- public question uploads
- public question authoring
- per-user saved content libraries
- horizontal scaling across multiple app instances
- native Android or iPhone apps

## Capacity Assumptions For Hosted v1

- design for about 2 active rooms at once
- usually fewer than 20 players per room
- single process
- in-memory room state is acceptable

These assumptions should influence implementation simplicity decisions.

## Target Architecture

## Top-Level State

Current state is centered around one global `GameState`. That model must be
replaced with a room registry.

Target top-level server state:

- `rooms: HashMap<String, RoomState>`
- `clients: ...` connection registry with room scoping
- `owner_index: HashMap<String, String>` mapping `owner_token -> room_code`
- runtime/static asset references
- cleanup task coordination
- public base URL configuration

## New Core State Types

### `RoomState`

Represents a room that may host multiple games over its lifetime.

Suggested fields:

- `room_code`
- `room_title`
- `owner_token`
- `room_status`
- `created_at`
- `last_activity_at`
- `launched`
- `players`
- `blocked_names`
- `clear_blocks_on_new_game`
- `pack_catalog_snapshot`
- `selected_pack_files`
- `settings`
- `current_game`

### `GameSessionState`

Represents one active or ended game inside an open room.

Suggested fields:

- `status`
- `questions`
- `shuffled_question_ids`
- `total_rounds`
- `completed_rounds`
- `current_round`
- score/leaderboard state
- any current-game moderation flags that should reset per game

### `RoomSettings`

Room-default or game-default configuration, likely derived from current
`GameState` fields:

- `speed_bonus_enabled`
- `hide_scores_until_end`
- `powerups_enabled`
- `response_seconds`
- `auto_issue_enabled`
- `auto_issue_delay_secs`
- `total_rounds`

### `PackCatalogEntry`

Friendly room-local snapshot metadata for one available server-side pack.

Suggested fields:

- `file_name`
- `display_name`
- `category`
- `question_count`
- `questions`

The first hosted implementation can derive display metadata from current file
names and pack content. This does not require a database.

## Lifecycle Separation

Room lifecycle and game lifecycle must be separate.

### Room lifecycle

- `setup`
- `open`
- `closed`
- `expired`

### Game lifecycle within an open room

- `lobby`
- `in_round`
- `round_result`
- `ended`

This separation is required because one room must be able to host multiple games
until it is explicitly closed or expires.

## API Direction

The current admin login flow should be removed. Hosted APIs should become
room-scoped and owner-token based.

Suggested public/API shape:

- `GET /`
- `POST /api/rooms/create`
- `POST /api/rooms/resume`
- `POST /api/rooms/launch`
- `POST /api/rooms/join`
- `GET /api/rooms/:code/public`
- `POST /api/rooms/:code/owner/settings`
- `POST /api/rooms/:code/owner/packs`
- `POST /api/rooms/:code/owner/start_game`
- `POST /api/rooms/:code/owner/end_game`
- `POST /api/rooms/:code/owner/close_room`
- `POST /api/rooms/:code/owner/kick_player`
- `POST /api/rooms/:code/owner/unban_name`
- `GET /api/qr.svg?text=...`

Exact route names may change, but the important constraint is:

- public actions must require only public room information
- owner actions must require owner-token authentication

## WebSocket Direction

Current WebSocket handling assumes one global game and client id addressing.
Hosted mode must make room scoping explicit.

Possible direction:

- one WebSocket endpoint with an initial join payload containing:
  - `room_code`
  - `role`
  - `player_id` or `owner_token`

Alternative:

- separate owner/player endpoints per room

Either approach is acceptable, but every pushed event and state broadcast must
become room-scoped.

Hosted mode should use a configured public base URL rather than LAN IP
detection.

Likely runtime behavior changes:

- disable auto-opening the admin browser in hosted environments
- stop generating LAN URLs for QR code/player join
- prefer an environment variable such as a public base URL for share links

The app should still support local binary execution where useful, but hosted mode
becomes the primary architecture.

## Phased Implementation Plan

## Phase 1: Server Architecture Refactor

Goal:
Replace single-room global server state with a room registry while preserving
existing gameplay logic as much as possible.

Tasks:

- Introduce `RoomState`, `GameSessionState`, and room registry structures.
- Move current single `GameState` assumptions behind room-scoped access.
- Make WebSocket and state broadcast logic room-aware.
- Keep current gameplay rules intact.

Exit criteria:

- server can manage more than one room in memory
- room-scoped game state is isolated

## Phase 2: Ownership And Room Lifecycle

Goal:
Replace admin login with room-owner token flow and add expiration/closure.

Tasks:

- Create room API with generated code and owner token
- owner-token resume API
- room activity tracking
- inactivity expiration cleanup
- close room API
- one-active-room-per-browser behavior in UI

Exit criteria:

- room owner can create, resume, and close a room without admin passcode
- expired rooms are cleaned up

## Phase 3: Public Homepage And Wizard

Goal:
Replace the old admin/player split entry model with a shared public homepage and
room setup flow.

Tasks:

- homepage with `Create Room`, `Resume Room`, `Join Room`
- room creation wizard
- room title validation
- pack-selection validation
- settings validation
- launch step generating room code and QR/share link

Exit criteria:

- non-technical user can create and launch a room from the public homepage

## Phase 4: Room Reuse And Dynamic Updates

Goal:
Support multiple games per room and room-local configuration persistence.

Tasks:

- support `End Game` without closing room
- preserve packs/settings while room remains open
- start a new game inside the same room
- apply pack/settings changes next round only

Exit criteria:

- a room can host multiple games without recreation

## Phase 5: Moderation And Player Flow Polish

Goal:
Add operational controls and complete hosted room behavior.

Tasks:

- kick player
- blocked-name policy and manual unban
- late-join waiting behavior
- between-game waiting screens
- no-score end-game message

Exit criteria:

- host has enough controls to manage a public room cleanly

## Phase 6: Hosted Deployment

Goal:
Deploy a working hosted version to a managed platform.

Tasks:

- add hosted runtime configuration
- configure public base URL
- disable local browser auto-open for hosted deployment
- verify WebSockets on platform
- run smoke test with 2 concurrent rooms

Exit criteria:

- hosted prototype is accessible publicly
- owner can create room from phone browser
- players can join from phones

## Suggested Near-Term Implementation Order

To reduce risk, the next coding sequence should be:

1. split current monolithic server logic into clearer room/game units
2. introduce room registry
3. migrate APIs to room scope
4. add owner token and resume
5. replace homepage/admin flow
6. deploy hosted prototype

## Known Risks

- `server/src/main.rs` is currently monolithic, so refactoring pressure will be
  high unless logic is first broken into smaller units.
- Current event and broadcast flow assumes one global room and must become
  room-scoped everywhere.
- Current question-pack persistence assumptions should be removed or reshaped,
  since selection becomes per-room rather than server-global.
- Public hosting requires stronger separation between public and owner actions
  than the current LAN-oriented trust model.
